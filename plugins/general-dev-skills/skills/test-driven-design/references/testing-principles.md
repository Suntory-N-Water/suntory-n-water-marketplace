## 「観察可能な振る舞い」と「実装の詳細」の見分け方

リファクタリングへの耐性を備えたテストを書くには、この2つを正しく見分ける能力が不可欠。

### 定義

すべてのプロダクションコードは、クライアント(そのコードを呼び出して使う側のコード)から見て以下のどちらかに分類される。

- **観察可能な振る舞い**: クライアントが自身の目標を達成するために直接的に利用するコード。「クライアントが目標を達成するために使う公開された操作(メソッド)」か「クライアントが目標を達成するために使う公開された状態」のいずれか。
- **実装の詳細**: 上記に該当しない、すべてのコード。

### public であっても実装の詳細になりうる

アクセス修飾子を public にして公開されていても、クライアントの目標達成に直接関わらない内部ロジックは「実装の詳細の漏洩」。

**具体例1：メソッドからの漏洩** — User クラスに「名前は50文字以内に収める」というルールがあり、文字列を切り詰める `NormalizeName` メソッドが public になっていた場合。クライアント(UserController)の目標は「ユーザの名前を変更すること」であり、「50文字以内に収めること」は内部の制約。したがって `NormalizeName` は public であっても実装の詳細であり、観察可能な振る舞いは「名前の変更(Name プロパティの更新)」だけ。

**具体例2：状態からの漏洩** — MessageRenderer クラスが、内部で使う補助的描画クラスのコレクション `SubRenderers` を public で保持していた場合。クライアントの目標は「HTMLを描画させること(Render メソッドの呼び出し)」であり、内部でどの補助クラスを使っているかはクライアントには関係ない。

### 漏洩の見分け方

「クライアントが1つの目標を達成するために、対象クラスの複数メソッドを順番に呼び出す必要があるか？」を確認する。Yes なら実装の詳細を漏洩している。理想は「1つの操作で目標を達成できる」こと。

### 不変条件の侵害とカプセル化

**不変条件(invariant)** — システム内で常に真でなくてはならない条件。実装の詳細が public に漏洩していると、クライアントが正しい手順を無視してデータの整合性を壊せてしまう。

「きちんと設計された API」の条件：
1. 公開 API が観察可能な振る舞いと完全に一致
2. 実装の詳細がすべて private として隠蔽
3. 「尋ねるな、命じよ(Tell, Don't Ask)」の原則に従う

## テスト手法の応用

### 出力値ベース・テストを実現する関数型アーキテクチャ

出力値ベース・テストを適用するには、テスト対象が副作用を持たない純粋関数である必要がある。ビジネス・ロジックと副作用を分離する「関数型アーキテクチャ」を導入する。

| 構成要素 | 役割 | テスト方法 |
|---|---|---|
| 関数的核(Functional Core) | すべてのビジネスにおける決定を下す。副作用を一切持たず、純粋関数として実装 | 出力値ベースの単体テスト |
| 可変殻(Mutable Shell) | 関数的核に渡す入力データを集め、関数的核が下した決定に基づいた副作用を実行 | テストケース数が少ない統合テスト |

### 値オブジェクトの活用

値オブジェクト(個別の識別性を持たず、内容によってのみ識別されるオブジェクト)を使うと、状態ベース・テストの品質が向上する。

- **コード量の削減**: 各プロパティを1行ずつ確認する代わりに、期待値となる値オブジェクトを1つ生成し、実際の結果と丸ごと比較するだけで済む(例：`expect(result).toEqual(expectedComment)`)。
- **安全性**: 不変(immutable)であるため、テストケース内でどのように扱っても状態が変わらず、他のテストケースに影響しない。

## Humble Object パターンとアーキテクチャ

### Humble Object パターン

過度に複雑なコードをテスト可能な形に分割するための設計パターン。

1. 複雑なビジネス・ロジックを別のクラスとして抽出 → 依存を持たない「ドメイン・モデル / アルゴリズム」(単体テストが容易)
2. ロジック抽出後に残ったクラス → ロジックをほぼ持たない質素な「コントローラ」(統合テストで検証)

コードの深さ(複雑さ)と広さ(協力者の数)が分離され、過度に複雑なコードが消滅する。

### ヘキサゴナル・アーキテクチャ

システム全体に Humble Object パターンを適用したアーキテクチャ設計。

| 層 | 役割 | テスト方法 |
|---|---|---|
| ドメイン層 | ビジネス・ロジックを担い、プロセス外依存から完全に隔離 | 単体テスト |
| アプリケーション・サービス層 | ドメイン層と外部アプリケーションとの連携を指揮 | 統合テスト |

## 古典学派とロンドン学派

### 違い

| 観点 | 古典学派(デトロイト学派) | ロンドン学派(モッキスト) |
|---|---|---|
| 隔離の対象 | テスト・ケース自体を他のテスト・ケースから隔離 | テスト対象のクラスを協力者オブジェクトから隔離 |
| 「単体」の定義 | 1単位の振る舞い(複数のクラスにまたがることもある) | 1つのクラス |
| テスト・ダブルの対象 | 共有依存のみ | 不変依存を除くすべての依存 |
| プライベート依存の扱い | 実物のまま使用 | テスト・ダブルに置き換える |

### 推奨：古典学派

古典学派のスタイルが推奨される。理由：

1. **ロンドン学派はリファクタリングへの耐性が低い** — クラス間のやり取りをすべてモックで検証するため、実装の詳細と極めて強く結びつく
2. **検証の焦点がずれている** — 1つのクラスという細かな粒度で検証しても、ビジネスの物語が伝わらない
3. **設計の欠陥をモックで隠蔽してしまう** — 複雑すぎる依存関係は設計の誤りを示唆するが、モックで切り離すと問題を隠蔽するだけ
4. **バグ特定の優位性は大きな差にならない** — こまめにテストを実行していれば「最後に修正した部分」にバグがあることは明らか

## アンチパターン

### private メソッドのテスト

**避けるべき**: テストのために private メソッドを public に変更して直接テストすること。

**正しい対処法**:
- 基本：公開されている API の一部として間接的に検証する
- private メソッドが複雑で公開 API から十分に検証できない場合：「抽象化の欠落」が起こっている可能性が高い。複雑なロジックを別のクラスとして抽出し、その公開メソッドとしてテストする
- 例外：O/Rマッパーが呼び出す private コンストラクタなど、実質的に観察可能な振る舞いの一部であるものは直接テストしても問題ない

### テストのために private な状態を公開すること

**避けるべき**: テストの確認フェーズで内部状態を検証したいがために、隠蔽すべき private な状態を public に変更すること。

**正しい対処法**: クライアントは内部の「状態」そのものではなく、状態変化がもたらす「結果」を気にしている。テストでは状態を直接見るのではなく、状態変化がもたらす最終的な結果(観察可能な振る舞い)を検証する。

### テストコードへのドメイン知識の漏洩

**避けるべき**: プロダクションコードに含まれる計算ロジック(例：`const expected = value1 + value2`)を、期待値を算出するためにテストコード内にそのまま複製すること。

**正しい対処法**: 期待値となる具体的な値(リテラル)をテストコードに直接ハードコードする。テスト対象のコードとは異なる方法(手計算やドメインエキスパートとの確認など)で事前に導き出した結果と比較する。

### 日時などの環境コンテキストの扱い方

**避けるべき**: `Date.now()` のような現在日時を、静的なメソッドやプロパティで取得・切り替えできるようにする「環境コンテキスト(ambient context)」パターン。

**正しい対処法**: 現在日時はメソッドの引数やコンストラクタを通じて「明示的な依存」として注入する。サービス(インターフェース)として注入するよりも「値」そのものとして注入する方がシンプル。
